# shopflow-core 目录说明

> 适用范围：`shopflow-core` 及其子目录，除非被更深层 `AGENTS.md` 覆盖。

## 1. 本目录职责

- 目录名称：`shopflow-core`
- 主要职责：承载跨模块通用基础设施与核心支撑能力，包括配置、鉴权、租户、事务、通知、存储、任务、过滤器、公共 handler 与三方能力封装。
- 不负责内容：不直接承载具体后台或小程序业务流程；不负责实体、mapper 和领域数据访问实现。

## 2. 复用与依赖边界

- 优先复用入口：本目录现有的 `config`、`satoken`、`tenant`、`storage`、`notify`、`tasks`、`transaction` 等封装。
- 优先复用入口：已有注解、切面、过滤器、handler、service 能力，避免在业务模块重复造轮子。
- 对外依赖边界：作为基础模块，应尽量保持稳定边界，避免反向依赖具体业务模块。
- 对外暴露边界：为 `shopflow-admin-api`、`shopflow-wx-api` 等上层模块提供稳定通用能力。

## 3. 编码约束

- 新增基础能力前，先确认现有封装是否可扩展，避免重复实现。
- 涉及鉴权、租户、事务、过滤器、任务调度、存储与通知时，优先兼顾全局影响和向后兼容。
- 公共能力接口要尽量稳定，避免频繁改动上层调用方式。
- 异常处理与日志输出要可追踪，不能吞错导致上层定位困难。

## 4. 测试与验证

- 当前目录修改后，优先验证：受影响模块编译通过、关键公共能力联调、配置加载与异常链路。
- 高风险点：鉴权拦截、租户隔离、事务边界、存储实现切换、消息通知、定时任务和三方服务封装。
- 变更后需要同步检查：上层调用点、配置文件、环境变量、部署说明和相关文档。

## 5. OpenSpec 衔接

- 涉及当前目录的功能新增、行为变更、主流程调整时，默认先走 OpenSpec。
- `apply` 完成后自动进入 `review`。
- `review` 完成后停止等待用户确认，再决定是否 `archive`。
- 如果归档结论只影响当前目录，优先沉淀到当前 `AGENTS.md`。
