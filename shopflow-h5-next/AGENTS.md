# shopflow-h5-next 目录说明

> 适用范围：`shopflow-h5-next` 及其子目录，除非被更深层 `AGENTS.md` 覆盖。

## 1. 本目录职责

- 目录名称：`shopflow-h5-next`
- 主要职责：提供基于 uni-app、Vue 3、TypeScript 的 H5 前台新实现，承载商城新风格页面、交易链路、个人中心页面、H5 兼容层与用户侧页面重构。
- 不负责内容：不直接承担后端业务规则落地，不替代 `shopflow-wx-api`；不把旧 `shopflow-h5` 的历史页面结构直接当成这里的实现约束。

## 2. 复用与依赖边界

- 页面数据优先复用：`src/entities/*/api.ts`
- 兼容逻辑优先复用：`src/shared/compat/*`
- 请求层优先复用：`src/shared/request/*`
- 视觉 token 优先复用：`src/shared/ui/tokens.css`
- 页面跳转与平台封装优先复用：`src/shared/platform/*`
- 若只是页面结构或样式重做，优先在页面内或对应 `*-utils.ts` 中收敛，不要先做过度组件化。

## 3. 页面视觉改造工作流

- 涉及页面换肤、布局重做、视觉统一、交互层级重排、个人中心/订单/购物车这类用户可见 UI 方案调整时，默认先经过根规则中的 `openspec-superpowers-orchestrator`。
- 若本次工作本质是“选 UI 方向”或“定页面风格方案”，应优先让编排入口路由到 `superpowers:brainstorming`，不要跳过设计确认直接改真实页面。
- 对 `shopflow-h5-next` 的页面视觉改造，除非用户明确要求直接实现，否则默认采用下面这套协作方式：
  - 先做一个本地可预览的 UI 方案页或 mock 页面
  - 在应用内浏览器里给用户看预览
  - 让用户先选择方向或确认方案
  - 用户确认后，再改真实页面代码
- 同一批页面后续延展时，优先沿用用户已经选定的视觉方向，避免每页重新漂移。

## 4. 浏览器验证方式

- 改动 `shopflow-h5-next` 的页面 UI 后，优先使用 `Node REPL + Browser Use / 应用内浏览器` 做真实页面检查、刷新与结构确认，不要只靠读代码判断页面已生效。
- 这类检查包括但不限于：
  - 刷新当前 H5 页面
  - 查看真实渲染结果
  - 检查 DOM 结构或关键文案是否出现
  - 确认滚动区、底部栏、TabBar、跳转结果等页面行为
- 除非用户明确要求别的方式，否则本目录下的页面联调优先基于应用内浏览器当前标签页完成，而不是只用 shell 输出描述结果。

## 5. 编码约束

- 保持 `ShopFlow` 作为对外品牌，不把参考稿名称直接写入用户可见文案。
- 参考外部设计稿时，可以借结构、层级、配色气质，但不得伪造用户 ID、会员等级、销量、积分等当前无真实字段支持的数据。
- 页面重构时优先把展示文案、映射规则、排序规则收敛到对应页面的 `*-utils.ts` 并补测试。
- 影响购物车、结算、支付、订单、个人中心等真实行为时，样式重构不得破坏原有跳转、按钮入口或关键交互。

## 6. 测试与验证

- 页面展示逻辑改动优先补对应 `*.test.ts`。
- 至少运行与改动直接相关的前端测试，以及：
  - `cd shopflow-h5-next && npm run type-check`
- 对用户可见页面改动，补一次 H5 本地视觉检查，推荐地址格式：
  - `http://127.0.0.1:6257/#/pages/...`

## 7. 文档沉淀

- 若用户确认了新的长期协作方式、预览流程或页面设计决策，优先沉淀到本目录 `AGENTS.md`，而不是只留在一次性对话里。
