# shopflow-admin 目录说明

> 适用范围：`shopflow-admin` 及其子目录，除非被更深层 `AGENTS.md` 覆盖。

## 1. 本目录职责

- 目录名称：`shopflow-admin`
- 主要职责：提供 Vue 2 + Element UI 管理后台前端，承载后台页面、路由、状态管理、接口调用、表单交互和运营管理界面。
- 不负责内容：不承担后端业务规则落地，不替代 `shopflow-admin-api`；不在页面层堆叠过多不可复用的业务逻辑。

## 2. 复用与依赖边界

- 优先复用入口：`src/api`、`src/components`、`src/utils`、`src/store`、`src/router` 中已有封装。
- 优先复用入口：已有表格、表单、上传、富文本、分页、字典等通用组件。
- 对外依赖边界：页面层通过统一 API 封装与后端交互，避免在视图层散落重复请求和数据转换。
- 对外暴露边界：对管理后台用户提供稳定、一致的操作流程与交互反馈。

## 3. 编码约束

- 页面行为改动前，先确认是否已有同类页面模式可复用，保持后台交互一致性。
- 新逻辑优先收敛到 `api`、`utils`、`store` 或可复用组件，不把复杂业务判断长期堆在单个 `.vue` 文件里。
- 接口字段映射、表单校验、权限控制和异常提示要和后端行为保持一致。
- 尽量沿用现有 Vue 2、Element UI、路由和状态管理风格，避免局部引入完全不同的实现范式。

## 4. 测试与验证

- 当前目录修改后，优先验证：页面渲染、表单提交流程、权限显示、异常提示和接口联调。
- 高风险点：菜单权限、角色分配、商品与订单管理、租户配置、上传与富文本编辑、复杂筛选表格。
- 变更后需要同步检查：对应后端接口、路由、字典数据、静态资源与构建结果。

## 5. OpenSpec 衔接

- 涉及当前目录的功能新增、行为变更、主流程调整时，默认先使用根规则中的 `openspec-superpowers-orchestrator` 作为统一入口，由它判断 `brainstorming` / `openspec-explore` 与后续阶段。
- 除已明确可按 `lightweight` 处理的极小改动外，不要绕过 `openspec-superpowers-orchestrator` 直接进入实现。
- `apply` 完成后自动进入 `review`。
- `review` 完成后停止等待用户确认，再决定是否 `archive`。
- 如果归档结论只影响当前目录，优先沉淀到当前 `AGENTS.md`。
