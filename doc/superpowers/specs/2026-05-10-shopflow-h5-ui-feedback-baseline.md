# shopflow-h5-next UI 反馈基线

## 背景

本文档用于记录 `shopflow-h5-next` 在本轮重构过程中，用户已经明确提出并反复确认的 UI 问题与后续整改约束。

目标不是描述最终设计方案，而是沉淀一份“不要再犯”的视觉基线，后续所有 H5 页面都应默认遵守。

## 用户已明确提出的问题

### 1. 整体视觉过丑，偏离实用型商城 H5

用户明确反馈：

- UI 太丑
- 不要花哨
- 只要简单一点
- 只是参考，不要照抄

结论：

- 当前新 H5 不应走“轻奢、发光、柔雾、大圆角、大留白”的视觉路线
- 应回到更朴素、更克制、更像移动电商商城的风格

### 2. 字体偏大

用户多次明确指出：

- 字体大
- 文字大小也要修改

结论：

- 标题、正文、辅助文案都需要整体缩小一档
- 不允许用偏大的展示型标题去撑页面气势
- H5 页面优先保证信息密度和实用性，而不是视觉冲击

### 3. border-radius 过大

用户多次明确指出：

- border radius 太大
- border-radius 控制在 15 以下就行
- 这里还有特别多
- 他不是 14px，他是 28px

结论：

- 圆角必须作为本轮视觉整改的一级约束
- 即使源码值理论上小于等于 `15px`，只要视觉上仍显得过圆，也要继续压缩
- 大容器上的 `14px` 视觉上仍可能显得偏圆，不能只看数值，不看观感

### 4. margin / padding / line-height 偏松

用户明确提出：

- line-height 间距也统一调小
- margin 和 padding 的间距也要处理

结论：

- 页面当前“臃肿感”不只来自圆角，还来自过大的留白和偏松的排版节奏
- 需要统一压缩：
  - 页面区块间距
  - 卡片内边距
  - 按钮上下 padding
  - 文本行高

### 5. 页面中出现无意义英文眉标

用户明确提出：

- `Browse Domain` 为什么每个上边都要加这种英文

结论：

- `Browse Domain / Auth Domain / Trade Domain / User Domain` 这类领域英文标签对真实用户无价值
- 不应作为页面视觉的一部分存在
- 后续默认不再添加这类英文眉标

## 本轮已确认的视觉整改方向

### 总体方向

- 简单
- 朴素
- 实用
- 更像商城 H5
- 少装饰
- 少英文标签

### 不应继续出现的设计倾向

- 大面积柔光、发光、雾化背景
- 夸张的大圆角容器
- 偏展示型的大标题
- 过松的上下留白
- 领域分层式英文眉标
- 为了“设计感”牺牲信息密度

## 后续页面必须遵守的 UI 约束

### 1. 圆角约束

- 外层主卡片优先控制在 `10px`
- 内层信息块优先控制在 `8px`
- 小标签、辅助块优先控制在 `6px ~ 8px`
- 默认不再使用视觉上明显偏圆的胶囊态按钮
- 若某组件即便数值合规但视觉仍显偏圆，继续下调，不以“源码已经 <= 15px”作为结束条件

### 2. 字号约束

- 页面主标题应控制在更克制的移动端尺寸
- 卡片标题、区块标题、正文、辅助文案均应整体比当前常见模板小一档
- 不允许为了“看起来高级”而人为拉大字重和字号

### 3. 间距约束

- 页面外边距压缩
- section 间距压缩
- 卡片 padding 压缩
- 表单按钮高度压缩
- 行高统一向更紧凑方向调整

### 4. 文案与视觉语言约束

- 面向用户的页面默认使用中文
- 不再在卡片顶部添加无业务价值的领域英文标签
- 仅保留对用户有帮助的信息，不保留开发期领域占位文案

## 当前应继续跟进的整改范围

本轮已经被用户直接看到、并继续需要重点盯紧的页面包括：

- 首页
- 分类页
- 登录页

后续应按相同基线同步到：

- 商品详情
- 分类商品列表
- 购物车
- 结算页
- 个人中心
- 其它占位页和迁移中页面

## 执行原则

- 优先服从用户观感反馈，不和用户争辩“代码里已经是 14px”
- 遇到“数值合规但视觉仍然不对”的情况，以视觉结果为准
- 每次样式整改后，优先检查：
  - 是否还显得圆
  - 是否还显得大
  - 是否还显得松
  - 是否还有没必要的英文标签

## 一句话基线

`shopflow-h5-next` 当前 UI 基线应理解为：

“不要轻奢感，不要大圆角，不要大字号，不要大留白，不要领域英文标签，要简单、紧凑、像一个真正能用的商城 H5。”
