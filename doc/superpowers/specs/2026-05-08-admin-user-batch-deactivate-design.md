# 会员管理批量注销设计稿

## 背景

当前管理后台会员管理已经支持单条“注销会员”，实际行为是将会员状态置为“注销”，不做物理删除。  
但当管理员需要一次性处理测试账号、无效账号或批量治理会员状态时，只能逐条点击注销，效率较低。

本次需求是在保持现有单条注销语义、权限和数据结构不变的前提下，为会员管理补充“批量注销会员”能力。

## 已确认范围

- 仅支持当前表格页已勾选会员批量注销。
- 不支持跨分页勾选。
- 不支持按当前筛选结果整批注销。
- 仍然是状态注销，不做物理删除。
- 允许部分成功。
- 前端只展示汇总结果，例如“成功 X 条，失败 Y 条”。
- 已注销会员在列表中选择框禁用，不能被勾选参与批量注销。
- 未注销会员可参与批量注销。
- 复用当前单条“注销会员”的权限，不新增独立权限点。

## 方案对比

### 方案 A：后端新增批量注销接口，前端走多选批量提交

做法：

- 管理端会员列表增加多选列和“批量注销”按钮。
- 前端将选中会员 id 列表一次性提交给后端批量接口。
- 后端逐条执行注销，统计成功数和失败数，返回汇总结果。

优点：

- 符合项目现有商品、通知、专题等批量操作模式。
- 更容易统一返回成功/失败汇总结果。
- 后续如果要扩展失败明细或操作日志，后端接口更容易承载。

缺点：

- 需要补一个新接口和对应前端封装。

### 方案 B：前端循环调用现有单条注销接口

做法：

- 前端逐条调用现有 `/admin/user/delete` 接口，并在前端自己统计结果。

优点：

- 表面改动较少。

缺点：

- 请求次数多，批量体验差。
- 结果汇总逻辑分散在前端，不利于后续扩展。
- 与项目现有批量接口模式不一致。

## 推荐方案

采用方案 A。

原因：

- 更符合当前项目管理端的批量操作实现习惯。
- 更容易稳定支持“允许部分成功 + 汇总提示”的需求。
- 不改变单条注销语义，只是在其上增加批量入口。

## 设计结论

### 前端交互

- 在会员管理列表中增加多选列。
- 已注销会员的多选框置灰禁用。
- 在查询区或工具栏增加“批量注销”按钮。
- 当未选择任何会员时，点击按钮直接提示“请至少选择一条未注销会员记录”。
- 当已选择记录时，弹出确认框，确认后发起批量注销请求。
- 操作完成后展示汇总提示，例如“批量注销完成，成功 8 条，失败 2 条”。
- 操作完成后刷新当前列表，并清空当前选中状态。

### 后端接口

- 新增会员批量注销接口，路径建议为 `/admin/user/batch-delete` 或与当前语义对齐的批量注销路径。
- 入参复用 `IdsBody` 风格，接收 `ids` 集合。
- 逐个查询会员并执行状态更新。
- 对于单条处理失败的会员，只累计失败数，不中断整个批次。
- 返回结果至少包含成功数量和失败数量，满足前端汇总提示。

### 权限策略

- 复用当前单条“注销会员”的权限语义。
- 前端批量按钮展示与后端接口权限校验保持一致。
- 不新增独立批量权限点，避免后台权限树额外扩张。

### 数据和兼容性

- 不新增数据库表、字段或迁移脚本。
- 不改变单条注销行为。
- 不影响会员历史订单、通知、分享、余额等既有关联数据查询。
- 不改动小程序端。

## 风险与边界

- 批量注销允许部分成功，因此前后端都不能默认“全成功”。
- 已注销会员不能参与选择，否则会让操作结果和管理员预期不一致。
- 如果批量接口和单条接口在状态更新逻辑上出现分叉，后续维护成本会变高，因此应尽量保持相同业务语义。
- 如果后端返回结构过于随意，后续扩展失败明细会困难，因此本次即使前端只展示汇总，也建议返回结构化计数结果。

## 验证建议

- 当前页勾选多个未注销会员后可以完成批量注销。
- 已注销会员的勾选框为禁用状态。
- 未选择任何记录时不能发起请求。
- 批量注销完成后可以看到成功数和失败数提示。
- 单条注销能力在改动后仍保持可用。
