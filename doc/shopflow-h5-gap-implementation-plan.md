# shopflow-h5 缺失功能开发排期明细

## 1. 文档目的

这份文档不是做概览说明，而是给开发排期和任务拆分使用。

目标是回答 5 个问题：

1. `shopflow-wx` 已有而 `shopflow-h5` 缺失的功能有哪些
2. 每个功能在小程序端对应哪些页面
3. H5 当前是否已有部分能力或接口基础
4. 每个功能依赖哪些接口、平台能力或额外方案
5. 应该按什么阶段来排期

## 2. 使用方式

适合以下场景直接使用：

- 产品和开发一起过缺口
- 技术负责人拆版本范围
- 前端评估 H5 工作量
- 后端确认哪些接口能直接复用

建议把每一行都视为一个候选开发项，而不是把“补齐 H5”当作一个大任务整体推进。

## 3. 现状判断原则

本文对“是否已具备”的判断，优先依据下面顺序：

1. H5 是否已有可访问页面与完整路由
2. H5 页面是否形成完整业务闭环
3. H5 是否仅保留 API 常量但没有页面实现

也就是说：

- 页面闭环已落地，才算“已有”
- 只有接口常量，没有页面闭环，不算“已有”

## 5. 分阶段建议

### 5.1 P1：优先补交易后链路

建议优先做：

- 售后申请
- 售后列表
- 售后详情
- 订单评价
- 商品评论列表

原因：

- 都直接挂在交易闭环后面
- 用户体验影响大
- 和当前 H5 已有订单、个人中心链路衔接最自然
- 复用后端现有接口的概率最高

### 5.2 P2：补用户留存和基础触达

建议第二阶段做：

- 足迹
- 消息列表

原因：

- 产品价值比较明确
- 复杂度低于完整聊天
- H5 可以先做浏览和删除，不必一步到位复刻小程序所有体验

### 5.3 P3：补互动和运营能力

建议第三阶段做：

- 聊天
- 动态
- 赏金活动
- 分享邀请
- 提现/余额
- 品牌入驻

原因：

- 这部分业务跨度大
- 更依赖微信生态或浏览器替代方案
- 前端工作量和交互方案不确定性更高

### 5.4 P4：补高耦合或价值待确认能力

建议最后评估：

- 品牌订单管理
- 二维码授权登录
- 管理员入口

原因：

- 这几项要么偏商家侧
- 要么偏微信特定场景
- 要么在 H5 上产品价值不一定成立

## 6. 各开发项明细

## 6.1 售后申请

### 小程序现状

- 页面：`pages/ucenter/aftersale/aftersale`
- 主要能力：
  - 从订单详情进入
  - 拉取订单详情
  - 选择退款类型
  - 填写退款原因和说明
  - 上传凭证图片
  - 提交售后申请

### H5 当前状态

- 没有售后申请页面
- 订单页只有“退款”动作，没有完整售后流程
- 个人中心只有退款列表壳子，没有真实售后闭环

### 可复用基础

- 订单详情页已存在
- 订单列表已存在
- H5 已有通用表单组件、弹窗和请求层

### 涉及接口

- `order/detail`
- `aftersale/submit`
- `storage/upload`

### 主要实现点

- 在订单详情页增加“申请售后”入口
- 新增 H5 售后申请页
- 支持退款类型选择、原因说明、凭证上传
- 提交成功后跳转售后列表或详情

### 风险和注意点

- H5 目前虽然有 `storage/upload` 常量，但上传流程未在这条链路现成打通
- 需要明确 H5 售后凭证是否与小程序使用同一上传协议

## 7. 能直接复用的接口与基础

以下能力较大概率可以直接复用当前后端接口，只需要 H5 新增页面和交互：

- 售后申请、列表、详情
- 订单评价
- 商品评论列表
- 足迹
- 消息列表
- 品牌入驻

原因是这些接口在小程序侧已明确存在，且业务语义比较稳定。

## 8. 需要先确认方案的能力

下面这些能力不建议直接按“小程序照搬 H5”推进，应该先做方案确认：

- 聊天
  - 需要确认 H5 WebSocket 协议、鉴权方式、断线重连策略
- 动态
  - 需要确认 H5 是否真的需要整套社区/互动模块
- 分享邀请
  - 需要确认 H5 的分享交互方案
- 提现/余额
  - 需要确认 H5 的提现目标和安全校验
- 二维码授权登录
  - 需要确认 H5 业务上是否必要
- 品牌订单管理
  - 需要确认是否应该归属 H5 前台

## 9. 建议的最小可执行版本

如果只做一版最小补齐，建议范围控制在下面 5 项：

1. 售后申请
2. 售后列表
3. 售后详情
4. 订单评价
5. 商品评论列表

这 5 项能明显补上 H5 在交易后链路上的短板，也最容易和当前已有 H5 订单体系衔接。

## 10. 建议下一步动作

如果要继续往实施走，建议下一步不是直接开改，而是先做一份真正用于编码的分解清单：

1. 为 P1 五项能力逐项确认 H5 页面入口
2. 为每项能力确认接口返回结构是否和小程序现用结构一致
3. 确认 H5 上传组件是否复用现有能力
4. 决定是新增页面还是改造历史遗留页面
5. 再进入具体排期和开发
