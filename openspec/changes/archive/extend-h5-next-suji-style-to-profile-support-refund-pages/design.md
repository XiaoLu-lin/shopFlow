# 设计：extend-h5-next-suji-style-to-profile-support-refund-pages

## Overview

本次变更延续用户已确认的 Batch B 预览，不再重新发散新的页面风格分支。核心策略是：

- 沿用 `ShopFlow` 现有灰蓝主题 token
- 继续使用 `8px` 卡片、按钮和输入框圆角
- 继续使用轻量 mini hero 作为页面识别区
- 让资料、服务、反馈和售后页面都回到同一套轻量用户页结构

## Shared Visual Rules

### 1. 颜色与层级

- 主色继续使用：
  - `--sf-color-brand`
  - `--sf-color-brand-light`
  - `--sf-color-brand-soft`
- 页面背景继续保持浅灰蓝到白色的轻渐变。
- 强调状态以品牌色、成功色和弱灰色区分，不重新引入暖色或高饱和色主导页面。

### 2. 圆角与阴影

- 卡片、面板、输入框、底部按钮统一使用 `8px` 圆角体系。
- 阴影继续使用轻量 `var(--sf-shadow-soft)`，状态按钮或 hero 使用已有品牌阴影，避免厚重悬浮感。

### 3. Hero 结构

- 每页顶部继续保留轻量 hero：
  - 左侧小字母 / 小头像块
  - 右侧眉文、标题、简短描述
- hero 只承担页面识别与轻情绪承接，不占据过大首屏。

### 4. 文案原则

- 对外继续使用 `ShopFlow`，不暴露 `Suji` 或内部设计命名。
- 不保留“兼容旧站”“先接回”“后续补齐”之类迁移期文案。
- 保持直白、轻量、生活化，但不过度修辞。

## Page Architecture

### 资料设置链路

#### 资料设置页

- 使用小 hero 承接“资料设置”页身份。
- 主体改为紧凑服务清单式结构，依次展示：
  - 昵称
  - 性别
  - 手机号
  - 密码设置
- 每项保留标题、当前值或简短说明、右箭头。
- 底部保留退出登录按钮。

行为保持：

- 昵称、手机号、密码入口仍跳转当前编辑页
- 性别继续保留当前点击切换逻辑
- 退出登录逻辑不变

#### 昵称 / 手机号编辑页

- 使用与地址编辑同风格的小 hero + 单字段轻表单。
- 主体只保留单个输入块和底部保存按钮。
- 输入框、按钮和页边距统一收紧。

行为保持：

- 昵称保存逻辑不变
- 手机号本地兼容更新逻辑不变

#### 重置密码页

- 使用轻量 hero + 分段表单。
- 手机号、验证码、新密码拆成独立轻卡片。
- 验证码按钮使用次级浅色按钮，不与主提交按钮争主次。

行为保持：

- 继续复用现有短信验证码与密码重置接口
- 倒计时逻辑不变

### 帮助 / 服务 / 反馈

#### 帮助中心

- 使用轻 hero + FAQ 列表。
- 单条 FAQ 表现为轻卡片，展开态显示答案。
- 展开按钮视觉从粗重符号改成更轻的标签或箭头感。

行为保持：

- 继续使用 `/issue/list`
- 当前展开 / 收起逻辑不变

#### 服务中心

- 使用轻 hero + 三条服务清单。
- 列表项使用与资料页一致的服务行模式。
- 联系客服弹层也统一为轻量底部卡片，而不是厚重弹窗。

行为保持：

- 联系客服弹层开关不变
- 跳转反馈页、帮助页逻辑不变

#### 意见反馈

- 使用轻 hero + 分段表单。
- 反馈类型改为小 chip 组，内容与联系方式为轻卡片输入块。
- 底部保留固定提交按钮。

行为保持：

- 继续复用当前提交接口与校验逻辑

### 售后链路

#### 售后列表页

- 使用轻 hero + 状态 chip + 紧凑售后记录卡。
- 记录卡保留：
  - 售后单号 / 订单号
  - 状态标签
  - 商品简卡
  - 退款原因 / 金额 / 时间
  - 查看售后 / 查看订单 / 撤销售后按钮

行为保持：

- 状态切换逻辑不变
- 查看订单 / 查看售后 / 撤销售后逻辑不变

#### 售后详情页

- 使用轻 hero + 分组信息卡：
  - 售后状态摘要
  - 商品信息
  - 售后信息
  - 凭证图片
- 底部保留查看订单与撤销售后按钮。

行为保持：

- 继续复用现有详情接口
- 继续支持图片预览和撤销售后

#### 申请售后页

- 使用轻 hero + 分段表单。
- 售后类型、原因、金额、说明、凭证上传分成多个轻卡片。
- 商品信息卡在最上方，底部保留固定提交按钮。

行为保持：

- 继续复用现有订单详情读取、上传凭证和提交售后申请逻辑

## Helper Strategy

- `page-display-utils.ts` 承担本批页面的 hero 与资料编辑页元信息，避免每页散落不同文案口径。
- `aftersale-utils.ts` 继续承担售后 tabs 和状态视觉语义，但状态 class 改为当前页面真实可用的样式命名。

## Testing Strategy

- 先为新增的展示辅助逻辑补测试，再落真实实现。
- 至少运行：
  - `cd shopflow-h5-next && pnpm vitest run src/pages/user/page-display-utils.test.ts`
  - `cd shopflow-h5-next && pnpm type-check`
- 实现完成后检查编译产物对应页面源码，必要时再做 H5 页面视觉核验：
  - `/#/pages/user/profile/index`
  - `/#/pages/user/help/index`
  - `/#/pages/user/service/index`
  - `/#/pages/user/feedback/index`
  - `/#/pages/user/refund/index?active=0`
  - `/#/pages/user/refund-detail/index?orderId=...`
  - `/#/pages/user/refund-apply/index?orderId=...`

## Risks and Mitigations

- 风险：资料页收轻后当前值信息不够显眼。
  - 处理：标题和当前值做清晰主次分层，当前值保留在右侧。
- 风险：售后状态 class 改动后颜色与旧页面不一致。
  - 处理：统一用页面内样式类接管，避免无效类名。
- 风险：帮助和服务页过轻后显得内容太空。
  - 处理：保留小 hero 和轻卡片分组，用间距和标签补足层次，但不新增无业务意义内容。
