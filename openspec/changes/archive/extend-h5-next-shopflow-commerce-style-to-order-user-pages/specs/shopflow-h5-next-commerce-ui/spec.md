# shopflow-h5-next 商城 UI 规格

## MODIFIED Requirements

### Requirement: ShopFlow H5 Next 必须提供灰蓝 ShopFlow 商城视觉基座

`shopflow-h5-next` MUST 在保留 `ShopFlow` 品牌的前提下，为商城主链路页面提供统一的灰蓝视觉基座，而不只覆盖首页、分类页和商品详情页。

#### Scenario: 第二阶段交易链路页面消费同一套视觉 token

- **前提** 第二阶段 UI refresh 已应用
- **当** 购物车、结算、订单、支付和个人中心页面渲染
- **则** 主品牌色、价格色、背景色、边框色和主要卡片圆角必须继续来自共享 ShopFlow token
- **且** 不得回退到第一阶段前的亮蓝旧视觉值

### Requirement: 购物车与结算页必须采用 ShopFlow 第二阶段交易布局

`shopflow-h5-next` 的购物车与结算页 MUST 在采用第二阶段交易布局时保留现有购物与下单行为。

#### Scenario: 购物车保留勾选与结算行为

- **前提** 购物车中存在商品
- **当** 用户浏览或编辑购物车
- **则** 页面必须提供勾选、全选、数量调整、删除和去结算操作
- **且** 底部结算栏必须清晰展示已选件数与合计金额

#### Scenario: 结算页保留 checkout flow context 契约

- **前提** 用户从购物车或立即购买进入结算页
- **当** 结算页加载与提交订单
- **则** 页面必须继续读取并写回现有 `AddressId / CartId / CouponId / UserCouponId` 上下文
- **且** 地址选择、优惠券选择、备注输入和提交订单行为不得回退

### Requirement: 订单与支付页必须在统一新风格下保留订单操作语义

订单列表、订单详情、支付页和支付结果页 MUST 呈现统一的第二阶段订单体验，同时保留现有交易状态和操作语义。

#### Scenario: 订单页继续支持关键操作

- **前提** 订单列表或订单详情数据可用
- **当** 用户浏览订单并执行操作
- **则** 页面必须继续暴露取消订单、删除订单、确认收货、申请售后和去支付等现有能力
- **且** 页面不得因为样式重排导致这些操作入口不可达

#### Scenario: 支付页继续遵循现有支付流程

- **前提** 用户从订单进入支付页
- **当** 用户发起支付并进入支付结果页
- **则** 页面必须继续保留微信内 JSAPI 支付、H5 支付跳转和支付结果页状态路由行为
- **且** 不得伪装尚未真正可用的支付方式为完整可支付能力

### Requirement: 个人中心必须纳入第二阶段 ShopFlow 用户侧风格

个人中心页 MUST 在第二阶段 UI refresh 后与商城主链路视觉一致，并保留现有用户服务入口。

#### Scenario: 个人中心保留现有服务入口

- **前提** 用户进入个人中心页
- **当** 页面渲染订单状态区和快捷入口区
- **则** 页面必须继续提供订单、购物车、收藏、地址、优惠券、售后、帮助、服务和设置等现有入口
- **且** 订单状态入口必须清晰区分待付款、待发货、待收货、待评价和售后相关语义

#### Scenario: 用户可见文案不再停留在迁移期

- **前提** 第二阶段页面渲染
- **当** 用户浏览购物车、结算、订单、支付或个人中心页面
- **则** 页面不得展示“已接回旧站”“下一批补齐”“兼容路径”等迁移期文案
- **且** 新文案必须保持 `ShopFlow` 品牌语义
