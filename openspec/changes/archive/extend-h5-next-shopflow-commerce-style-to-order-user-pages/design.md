# 设计：extend-h5-next-shopflow-commerce-style-to-order-user-pages

## 总览

本次设计延续第一阶段 `refresh-h5-next-shopflow-commerce-style` 的视觉结论，继续参考 `docs-suji.tar.gz` 中购物车、个人中心与交易链路页面的布局方式，但对外品牌、文案和业务语义保持 `ShopFlow`。

本次不新增后端字段，不改变路由契约，不重新设计支付或下单流程。实现策略是“保留行为、重塑层级”，即以现有接口、现有页面逻辑为行为基线，重写页面结构、视觉区块和状态呈现方式。

上游参考与约束：

- `doc/superpowers/specs/2026-05-15-shopflow-h5-next-commerce-style-design.md`
- `openspec/specs/shopflow-h5-next-commerce-ui/spec.md`
- `/Users/lhl/Desktop/code/docs-suji.tar.gz`

## 参考稿映射

本次第二阶段主要借鉴 `docs-suji/pages/README.md` 中以下页面：

- Cart：单列商品卡、圆形勾选、底部全选与结算栏。
- Profile：顶部用户头部、订单状态宫格、工具入口列表。
- Payment 相关：灰蓝主支付面板与清晰的结果状态反馈。

同时继续沿用参考稿的统一 token 倾向：

- 页面背景 `#F8F9FB`
- 卡片纯白
- 主色灰蓝 `#4A6FA5`
- 价格红 `#E8553D`
- 卡片/面板圆角以 `8px` 为主，局部大容器可使用略大圆角

由于第一阶段已经在 `shopflow-h5-next/src/shared/ui/tokens.css` 内把常规卡片圆角收敛到约 `8px`，本次不得回退到旧亮蓝风格，也不把参考稿中偏大的 `12px` 到 `16px` 大圆角照搬成全局默认。

## 页面范围与结构

### 购物车页

保留现有行为：

- 勾选/取消勾选
- 全选/取消全选
- 编辑态删除
- 数量加减
- 跳转结算

结构调整为：

- 顶部简洁标题区，可带购物车说明，但不再出现“已接回旧站”等迁移说明。
- 中间单列购物车卡片列表，图片、名称、规格、价格、数量和勾选更靠近参考稿布局。
- 编辑态与普通态共享卡片结构，只在数量区与删除入口上切换。
- 底部固定结算栏使用白底面板 + 品牌色结算按钮；总价使用统一价格色。

### 结算页

保留现有行为：

- 读取 checkout flow context
- 地址选择
- 优惠券选择
- 备注输入
- 提交订单

结构调整为：

- 顶部标题区改为交易型结算页头。
- 地址区卡片化，强调收货人、电话和完整地址。
- 商品列表使用统一的交易商品卡。
- 优惠券区采用横向 chip 或横向卡片。
- 金额汇总固定在页面底部摘要区或底部提交区，突出实付金额和提交按钮。

### 订单列表页

保留现有行为：

- 按状态切换订单列表
- 进入订单详情
- 取消、删除、确认收货、支付

结构调整为：

- 顶部订单标题区与状态 chips 统一成灰蓝体系。
- 订单卡内商品展示保持一单多商品结构，但强化状态、金额、操作按钮层级。
- 空态文案更中性，不再出现“等你下完一单后”这类偏调试式表述。

### 订单详情页

保留现有行为：

- 查看商品、收货、金额、物流
- 取消、删除、申请售后、确认收货、支付

结构调整为：

- 顶部状态摘要卡单独突出订单状态与关键金额。
- 商品信息、收货信息、订单信息、金额明细、物流信息拆分成更清晰的卡片层级。
- 底部操作按钮主次分明，品牌主操作与次操作分色。

### 支付页与支付结果页

保留现有行为：

- 微信支付优先
- 微信内 JSAPI 支付
- H5 支付跳转
- 支付结果页状态识别与自动跳转

结构调整为：

- 支付页突出订单号、应付金额和支付方式卡。
- 未真正可用的支付方式继续保留入口，但明确展示为后续能力，不伪装成可完整支付。
- 支付结果页按成功、取消、失败三种结果区分图形、颜色和文案，并保持订单列表回跳。

### 个人中心页

保留现有行为：

- 拉取用户概览
- 进入订单列表
- 进入购物车
- 进入收藏、地址、优惠券、售后、帮助、服务、设置等页面

结构调整为：

- 顶部用户头部采用更明显的品牌化层级，强化昵称、头像和设置入口。
- “订单状态”作为一组图标/数字入口卡片，优先对应待付款、待发货、待收货、待评价、售后。
- 快捷入口收敛为语义明确的服务卡片，减少“下一批补齐”式文案。

## 组件与复用策略

本次优先少量沉淀、避免过度抽象：

- 如果购物车、结算页、订单页都需要同类交易商品展示，可抽取轻量 `OrderGoodsCard` 或等价局部组件。
- 如果订单状态 chip、金额摘要或支付状态拷贝逻辑变复杂，可抽纯函数或小型样式块。
- 对仅在单个页面使用的结构，优先保持页面内实现，避免为了“复用”提前做大抽象。

## 数据与兼容边界

- 所有 API 继续复用现有 `entities/cart/api.ts`、`entities/order/api.ts`、`entities/user/api.ts`。
- 不改 checkout flow context 的 key 和写入行为。
- 订单、支付和售后入口继续沿用现有路由。
- 参考稿中的购物车角标、评价数、账号 ID、订单状态装饰等，如接口无直接字段，前端应使用保守文案或隐藏，不生成伪造数据。

## 测试策略

本次优先在以下位置补可测逻辑：

- 购物车页涉及的金额、勾选或底部摘要逻辑，若新增归一化处理则补 `features/cart` 测试。
- 订单/支付结果文案、订单状态展示规则如有新增 helper，则补 `features/order` 或 `pages/user` 测试。
- 个人中心快捷入口如果扩展了类型与跳转规则，应补 `entry-utils.test.ts`。

阶段验证：

- `cd shopflow-h5-next && npm run type-check`
- `cd shopflow-h5-next && npm run test`
- `cd shopflow-h5-next && npm run build:h5`
- H5 视口下人工检查购物车、结算、订单列表、订单详情、支付页、支付结果页和个人中心页。
