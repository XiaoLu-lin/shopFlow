# Proposal: align-litemall-vue-tenant-user-headers

## Why

当前导入的 `litemall-vue` 前台来自旧版 litemall H5 方案，请求协议和鉴权流程仍然基于历史实现，但当前 ShopFlow 用户侧后端已经演进为新的多租户与多登录态协议：

- 用户登录态通过 `X-ShopFlow-User-Token` 传递。
- 多租户上下文通过 `X-ShopFlow-TenantId` 传递。
- 首页初始化依赖 `appid` 生成租户 token，再进入后续用户侧接口链路。
- 当前 `/wx/auth/login` 已不再是旧商城普通账号登录，而是“已授权用户绑定管理员账号”的流程。

因此旧前台直接访问当前 `/wx/*` 接口时，会出现两组兼容问题：

1. 租户上下文兼容问题
   - 未携带 `appid` 时，首页初始化接口无法生成租户 token。
   - 未携带 `X-ShopFlow-TenantId` 时，大部分 `/wx/*` 接口会被租户过滤器拦截，返回“未找到授权租户”。

2. 鉴权语义兼容问题
   - 旧前台登录页仍按 `username/mobile/email + password` 方式调用 `/wx/auth/login`，会触发 `A0223`“用户未获得第三方登陆授权”。
   - 旧前台注册、找回密码、验证码和登录成功返回结构也仍沿用历史接口约定，与当前 ShopFlow `/wx/auth/*` 存在路径、参数或返回值差异。
   - 旧前台找回密码页面本身还存在未接线的空实现，无法仅靠 header 兼容恢复。

用户已确认将这两类问题合并为一次兼容治理，目标不是完全重做前台，而是在保持现有页面结构和主要路由不变的前提下，让导入的旧 `litemall-vue` 恢复可联调、可登录、可注册、可重置密码的基础能力。

## What

本次变更将扩展当前 change 范围，统一处理 `litemall-vue` 与当前 ShopFlow 用户侧后端之间的协议兼容：

1. 保留并完善前台租户初始化兼容。
2. 在首页初始化阶段传入可配置的 `appid`，获取并持久化当前环境可用的租户 token。
3. 在后续 `/wx/*` 请求中统一携带 `X-ShopFlow-TenantId` 和 `X-ShopFlow-User-Token`。
4. 为旧 H5 前台补齐账号登录兼容，恢复普通用户账号密码登录能力。
5. 为旧 H5 前台对齐注册、短信验证码、密码重置等 `/wx/auth/*` 接口。
6. 对齐旧前台登录成功后的返回结构、token 存储字段和登录态判断。
7. 补齐旧前台找回密码流程中的未实现页面逻辑。

本次变更预期会同时修改：

- `litemall-vue`
- `shopflow-wx-api`

本次变更不包含：

- 全量修复旧版 `litemall-vue` 与当前 ShopFlow `/wx/*` 的全部接口路径差异。
- 重写微信登录、管理员扫码授权、管理员绑定、后台登录等现有小程序专属流程。
- 修改 `shopflow-core` 现有租户鉴权规则去兼容旧头名称。
- 修改数据库结构、租户表模型或 token 协议。
- 为旧前台恢复“邮箱登录”这类当前后端并未提供稳定 C 端支持的历史分支。

## Impact

受影响模块：

- `litemall-vue`
- `shopflow-wx-api`

受影响能力：

- 旧 H5 前台首页初始化
- 旧 H5 前台用户侧请求头兼容
- 旧 H5 前台普通用户登录
- 旧 H5 前台用户注册
- 旧 H5 前台短信验证码申请
- 旧 H5 前台密码重置

关键假设：

- 当前环境存在一个可用的前台 `appid` 来源，至少包含默认配置值 `1649067` 这一可用启动值。
- 旧前台仍通过 `src/utils/request.js` 统一发起绝大多数 `/wx/*` 请求，适合在请求层集中做双 header 与租户预热兼容。
- 当前后端现有 `register`、`reset`、`captcha/mobile` 等 C 端能力可以复用，但旧前台需要对齐其真实路径、参数结构和返回结构。
- 普通用户账号登录兼容应作为新的前台专用兼容入口补充到 `shopflow-wx-api`，而不是直接改写当前 `/wx/auth/login` 的管理员绑定语义。

主要风险：

- 登录前相关接口虽然可走租户白名单首包，但注册、重置密码、验证码等链路仍依赖有效租户上下文，必须保证前台先完成租户预热。
- 如果后端新增旧 H5 登录兼容入口时误复用或覆盖了现有 `/wx/auth/login`，可能破坏小程序管理员绑定链路。
- 旧前台局部页面除了协议问题，还存在自身未完成实现；若只修接口不补页面逻辑，用户仍会感知为“功能不可用”。
- 当前后端用户模型主要稳定支持手机号维度，旧前台历史 `email` 分支即使保留输入判断，也不能承诺完整可用。

预期收益：

- 让导入的旧前台从“只能匿名浏览部分页面”提升为“可匿名浏览 + 可登录注册 + 可找回密码”的基础可用状态。
- 保持现有 ShopFlow 后端核心租户与登录协议不倒退，同时为旧 H5 提供最小必要的兼容入口。
- 为后续逐步修复商品、购物车、订单等更深层接口差异建立稳定的鉴权和租户底座。
