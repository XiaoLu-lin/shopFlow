# Proposal: simplify-github-actions-build-pipeline

## Why

当前项目的 GitHub Actions 工作流同时承担了构建和自动部署两类职责，但自动部署节点目前并未实际投入使用：

- 服务器密钥、上传路径和重启脚本链路增加了额外维护成本。
- 未使用的远程部署步骤会放大 CI 失败面，也增加后续误触发部署的风险。
- 当前团队更明确的需求是保留前端构建与后端打包校验，而不是在 CI 中直接发版到服务器。

因此需要把现有 workflow 收敛为纯构建流水线，只保留前后端产物构建与归档能力。

## What

本次变更将精简 GitHub Actions 工作流：

1. 保留后端 Maven 打包 job。
2. 保留前端 Vue 构建 job。
3. 删除当前未使用的自动部署 job。
4. 保持现有触发方式不变，仍支持 `push` 到主分支和手动触发。
5. 保持构建产物上传为 artifact，便于后续人工下载或扩展独立部署流程。

本次变更不包含：

- 新增其他 CI 平台配置。
- 新增 Docker 镜像构建。
- 新增新的发布、灰度或回滚流程。
- 修改服务器部署脚本内容。

## Impact

受影响模块：

- `.github/workflows`

受影响能力：

- GitHub Actions CI/CD 流程

主要风险：

- 如果后续仍有人依赖当前 workflow 的自动上传服务器能力，删除 deploy job 后需要改为人工部署或单独设计新的发布流程。
- 如果 workflow 中有步骤隐式依赖 deploy job 的产物命名，需要确认保留的 build job 仍独立可用。

预期收益：

- 降低 CI 配置复杂度和失败面。
- 避免未启用的自动部署逻辑长期滞留在主流程中。
- 让流水线职责更清晰，聚焦“前端构建 + 后端打包”的持续集成目标。
