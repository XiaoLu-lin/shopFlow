# Proposal: add-admin-user-delete-and-batch-goods-delete

## Why

当前管理后台存在两个明显的运维缺口：

- 会员管理页面只能查询、编辑、查看分享与余额，不能直接注销会员，导致误注册、测试账号或无效账号需要额外走数据库或业务绕行处理。
- 商品列表支持单个删除，但不支持批量删除，管理员在清理测试商品、违规商品或历史无效商品时效率较低。

这两个能力都属于后台管理侧的常规操作，且前后端结构已经具备可复用的删除模式，适合通过最小改动补齐。

## What

本次变更将补充两项后台能力：

1. 在会员管理页面增加“注销会员”操作。
2. 在商品列表页面增加“批量删除”操作。

具体实现范围：

- 管理端前端增加对应按钮、勾选、确认弹窗与刷新逻辑。
- 管理端 API 封装增加会员注销与商品批量删除请求。
- 管理后台后端控制器增加会员注销接口与商品批量删除接口。
- 商品批量删除复用现有单个商品删除逻辑，避免重复实现商品级联删除细节。
- 保持现有单个商品删除功能不变。
- 保持当前页面的查询、导出、编辑、审核等既有能力不受影响。

本次变更不包含：

- 会员批量注销。
- 会员物理删除及其复杂关联清理重构。
- 商品批量删除的异步任务化或进度展示。
- 新增数据库表、字段或迁移脚本。

## Impact

受影响模块：

- `shopflow-admin`
- `shopflow-admin-api`

受影响能力：

- 后台会员管理
- 后台商品管理

主要风险：

- 会员如果做物理删除，可能影响关联数据展示或历史记录引用，因此本次改为状态注销，避免破坏既有依赖链路。
- 商品批量删除必须复用现有单删逻辑，否则容易漏掉团购、赏金等关联清理。
- 管理端按钮权限和后端权限标识需要保持一致，避免前后端能力不匹配。
- 批量删除接口如果部分成功、部分失败，前端提示与后端返回必须一致，避免管理员误判。

预期收益：

- 降低测试账号、脏数据、无效会员的后台停用与治理成本。
- 明显提升测试商品、违规商品和历史废弃商品的批量处理效率。
- 保持当前项目在管理端删除交互上的一致性，减少额外学习成本。
