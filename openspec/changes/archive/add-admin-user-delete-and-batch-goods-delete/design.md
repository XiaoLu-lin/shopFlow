# Design: add-admin-user-delete-and-batch-goods-delete

## Overview

本次改动只涉及管理端与管理后台接口，属于已有删除能力的补齐，不改动小程序端和数据库结构。

整体策略：

- 会员注销采用“前端单条操作 + 后端单条注销接口”的最小改动方式。
- 商品批量删除采用“前端多选 + 后端批量接口”的交互方式。
- 商品批量删除后端不重新发明删除流程，而是循环复用现有 `goodsCoreService.goodsDelete(id)` 单删逻辑，确保团购、赏金等关联清理保持一致。

## Admin User Deactivate

前端：

- 在会员管理列表操作列新增“注销”按钮。
- 点击后弹出确认框，确认后调用注销接口。
- 注销成功后提示并刷新当前列表。
- 删除按钮沿用当前列表页操作列风格，避免引入新的交互模式。

后端：

- 在 `AdminUserController` 新增 `/admin/user/delete` 接口。
- 沿用当前控制器的权限声明和返回结构。
- 接口保持现有删除命名，但实际行为改为将会员状态置为注销状态，不做物理删除。
- 如果会员不存在或更新失败，则返回删除失败响应，保持与项目其他删除接口一致。

## Admin Goods Batch Delete

前端：

- 在商品列表增加多选列。
- 在顶部工具栏增加“批量删除”按钮。
- 未选中任何商品时给出提示。
- 选中后弹出确认框，确认后调用批量删除接口。
- 删除成功后提示并刷新列表，同时清空当前选中状态。
- 批量删除按钮样式、禁用态和提示文案尽量对齐现有通知、专题等页面。

后端：

- 在 `AdminGoodsController` 新增 `/admin/goods/batch-delete` 接口。
- 接收 `IdsBody` 中的 `ids` 集合。
- 遍历 `ids`，逐个调用现有 `goodsCoreService.goodsDelete(id)`。
- 任一删除失败时返回失败响应，避免前端误判全部成功。
- 不绕过 `goodsCoreService.goodsDelete(id)` 直接调用底层 service 删除，避免漏掉团购、赏金和其他后续清理逻辑。

## Failure Handling

- 前端在取消确认框时，不发送请求、不刷新页面。
- 前端在未勾选任何商品时，直接提示“请选择至少一条记录”。
- 后端遇到空 id、会员不存在、删除结果异常或底层删除失败时，返回现有失败响应结构，不新增特殊返回协议。

## Permission Strategy

- 会员注销使用独立权限点，便于和会员编辑、会员详情分开控制。
- 商品批量删除使用独立权限点，避免默认等同于单个删除。
- 前端按钮展示和后端 `SaCheckPermission` 路径保持一一对应。

## Permissions

为保持与现有后台约定一致：

- 会员注销新增独立权限标识。
- 商品批量删除新增独立权限标识。
- 前端按钮展示沿用现有 `v-permission` 约定。

## Verification

- 会员列表能够显示注销按钮，并可完成确认注销与列表刷新。
- 商品列表能够多选并批量删除。
- 商品批量删除后，现有单删涉及的关联清理仍然生效。
- 未选择商品时，批量删除不会发送请求。
- 取消删除时不应触发接口调用。
- 单个商品删除能力在改动后仍保持可用。
