# Design: add-admin-user-batch-deactivate

关联设计稿：`doc/superpowers/specs/2026-05-08-admin-user-batch-deactivate-design.md`

## Overview

本次改动只涉及管理端与管理后台接口，属于现有单条会员注销能力的批量扩展，不改动小程序端、不改数据库结构。

整体策略：

- 前端采用项目已有批量操作交互模式：多选列 + 顶部批量按钮 + 确认框。
- 后端新增批量注销接口，接收选中会员 id 集合并逐个执行状态更新。
- 业务语义保持与现有单条注销一致，即将会员状态置为“注销”，不做物理删除。
- 批量操作允许部分成功，前端只展示汇总结果，不展示逐条明细。

## Admin User Batch Deactivate

前端：

- 在会员管理列表增加多选列。
- 对状态为“注销”的会员禁用勾选框。
- 在筛选区或工具栏增加“批量注销”按钮。
- 未选择任何未注销会员时，点击按钮直接提示，不发送请求。
- 选中记录后弹出确认框，确认后调用批量注销接口。
- 接口返回后展示汇总结果，例如“批量注销完成，成功 X 条，失败 Y 条”。
- 处理完成后刷新列表并清空当前选中状态。

后端：

- 在 `AdminUserController` 增加批量注销接口。
- 入参复用 `IdsBody`，接收会员 `ids` 集合。
- 遍历 `ids`，逐个查询会员并执行状态更新。
- 对已不存在、已更新失败或其他处理失败的会员累计失败数。
- 整个批次不因单条失败中断，最终返回成功数和失败数。

## State Handling

- 会员状态仍沿用现有枚举和字段语义。
- 批量注销只处理“未注销”会员。
- 已注销会员在前端不可勾选，因此正常链路下不会进入批量接口。
- 即使前端被绕过，后端仍应按失败或忽略策略稳定处理，避免抛出未受控异常。

## Permission Strategy

- 复用当前单条“注销会员”的权限语义。
- 前端批量按钮展示沿用当前会员注销权限控制方式。
- 后端批量接口权限校验与单条注销保持同一权限口径，不新增独立批量权限点。

## Failure Handling

- 前端取消确认框时，不发送请求，不刷新页面。
- 前端未选择任何可用记录时，直接提示并返回。
- 后端对单条处理失败只计入失败数，不影响其他会员继续处理。
- 前端收到结果后，不应按“全成功”文案提示，而应始终按照返回的成功数和失败数进行汇总提示。

## Verification

- 当前页可以勾选多个未注销会员并完成批量注销。
- 已注销会员的勾选框为禁用状态。
- 未勾选任何记录时，批量注销不会发送请求。
- 批量注销返回后，页面能展示成功数和失败数。
- 单条注销能力在改动后仍然可用。
- 列表刷新后，已注销会员状态展示正确。
