# Proposal: add-admin-user-batch-deactivate

关联设计稿：`doc/superpowers/specs/2026-05-08-admin-user-batch-deactivate-design.md`

## Why

当前管理后台会员管理已经支持单条“注销会员”，但在批量治理测试账号、无效账号或历史脏数据时，管理员仍需逐条点击处理，操作效率较低。

对于会员治理场景来说，批量注销属于现有单条注销能力的自然延伸，且项目中已有商品、通知、专题等后台批量操作模式，可以复用成熟交互和接口风格，以最小改动补齐会员管理能力。

## What

本次变更将在会员管理中增加“批量注销会员”能力。

具体范围：

1. 会员列表增加多选能力。
2. 已注销会员在列表选择框处禁用，不能参与勾选。
3. 管理端增加“批量注销”按钮、确认弹窗和结果提示。
4. 管理端新增会员批量注销 API 封装。
5. 管理后台新增会员批量注销接口，支持部分成功并返回成功数、失败数。
6. 批量注销后刷新当前列表并清空当前选中状态。

本次变更不包含：

- 会员物理删除。
- 跨分页勾选批量注销。
- 按当前筛选结果整批注销。
- 展示逐条成功/失败明细。
- 新增独立批量注销权限点。
- 数据库结构变更。

## Impact

受影响模块：

- `shopflow-admin`
- `shopflow-admin-api`

受影响能力：

- 后台会员管理

主要风险：

- 批量注销允许部分成功，前后端需要对“非全成功”结果保持一致表达，避免误导管理员。
- 已注销会员若仍可勾选，会造成重复操作与提示歧义，因此前端必须明确禁用。
- 复用单条注销权限后，前端按钮展示与后端权限校验必须一致，避免出现前端可见但后端无权的情况。

预期收益：

- 明显提升后台会员治理效率。
- 保持后台批量操作模式一致，减少管理员学习成本。
- 在不改变现有会员注销语义的前提下补齐管理能力。
