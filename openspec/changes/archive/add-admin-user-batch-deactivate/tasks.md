# Tasks: add-admin-user-batch-deactivate

关联设计稿：`doc/superpowers/specs/2026-05-08-admin-user-batch-deactivate-design.md`

## 1. 会员管理批量注销前端交互

- [x] 在会员管理列表增加多选列，并对已注销会员禁用勾选框。
  - 验收契约：
    - MUST 只有未注销会员可以被勾选参与批量注销。
    - SHALL 已注销会员的选择框在列表中呈禁用状态。
- [x] 在会员管理页面增加“批量注销”按钮、未选中提示、确认弹窗和完成后的列表刷新逻辑。
  - 验收契约：
    - MUST 未选择任何未注销会员时不发送请求。
    - SHALL 操作完成后刷新当前列表并清空当前选中状态。

## 2. 管理端接口封装

- [x] 为管理端用户 API 增加会员批量注销请求封装。
  - 验收契约：
    - MUST 以 `ids` 集合方式提交当前页已选会员 id。
    - SHALL 与后台批量注销接口路径和请求方法保持一致。

## 3. 管理后台批量注销接口

- [x] 为管理后台用户控制器增加会员批量注销接口，并逐个执行状态注销。
  - 验收契约：
    - MUST 保持会员注销语义为“更新状态为注销”，不得物理删除会员数据。
    - SHALL 允许部分成功，不因单条失败中断整个批次。
- [x] 为批量注销接口返回成功数和失败数，满足前端汇总提示需要。
  - 验收契约：
    - MUST 返回可直接用于“成功 X 条，失败 Y 条”提示的汇总结果。
    - SHALL 复用当前单条“注销会员”的权限口径，不新增独立权限点。

## 4. 联调与验证

- [x] 验证批量注销、已注销禁选、未选中提示、汇总提示和列表刷新是否符合预期。
  - 验收契约：
    - MUST 覆盖“部分成功”场景下的汇总提示。
    - SHALL 确认单条注销能力在改动后仍可继续使用。
