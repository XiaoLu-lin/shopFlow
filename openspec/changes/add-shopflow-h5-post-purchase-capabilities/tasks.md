- [ ] 创建本次 change 的 proposal、design、tasks 和 change 内 spec。
- [ ] 补充 brainstorming 设计稿与 writing-plans 实施计划。

- tasks SHALL 明确 P1 范围与排除项，避免实现期范围膨胀。

## 2. 售后页面与入口补齐

- [ ] 新增 H5 售后申请页，并接通订单详情页中的“申请售后”入口。
- [ ] 新增 H5 售后列表页，并在个人中心补齐入口。
- [ ] 新增 H5 售后详情页，并完成列表到详情跳转。

验收契约：

- 用户在已登录状态下 MUST 能从订单详情进入售后申请页。
- 售后列表 SHALL 支持分页与状态展示。
- 售后详情 MUST 展示订单、商品、金额、状态、原因、说明和凭证信息。

## 3. 售后提交流程接通

- [ ] 接通 `order/detail`、`aftersale/submit`、`storage/upload` 到 H5 售后申请页。
- [ ] 实现售后类型、原因、说明和凭证上传校验。
- [ ] 处理提交成功、失败和未登录态反馈。

验收契约：

- 售后申请提交前 MUST 校验必要字段。
- 非“未收货退款”场景在需要凭证时，不得允许空图片直接提交。
- 提交成功后 SHALL 给出明确成功反馈并进入后续可查看路径。

## 4. 订单评价与商品评论补齐

- [ ] 新增 H5 订单评价页。
- [ ] 接通订单列表与订单详情中的“去评价”入口。
- [ ] 接通 `order/goods`、`order/comment`、`storage/upload`。
- [ ] 新增 H5 商品评论列表页，并在商品详情中补齐入口。
- [ ] 接通 `goods/comment/count` 与 `goods/comment/list`。

验收契约：

- 已具备评价资格的订单商品 MUST 能进入评价页并成功提交。
- 商品评论列表 SHALL 支持全部评论与有图评论切换。
- 评论失败时 MUST 给出可见错误提示，不得停留为空白或静默失败。

## 5. 复核与验证

- [ ] 复核旧 `refund-list` 的去留策略，避免保留误导性空壳页面。
- [ ] 完成 H5 关键链路手工验证与最小构建验证。
- [ ] 补充 review，记录已补齐范围、残余风险和未纳入本次的后续能力。

验收契约：

- review SHALL 明确本次已完成的售后与评论闭环范围。
- review MUST 单独标注足迹、消息、动态等未纳入能力为后续事项。
- 在未提供最新验证证据前，不得声称本次能力已完成可归档。
