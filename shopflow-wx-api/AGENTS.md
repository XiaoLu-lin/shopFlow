# shopflow-wx-api 目录说明

> 适用范围：`shopflow-wx-api` 及其子目录，除非被更深层 `AGENTS.md` 覆盖。

## 1. 本目录职责

- 目录名称：`shopflow-wx-api`
- 主要职责：提供微信小程序与 C 端用户侧后端接口，承载首页、商品、购物车、订单、用户、优惠券、动态、消息等用户侧业务能力。
- 不负责内容：不承担通用基础设施的底层实现，不替代 `shopflow-core`；不负责底层实体与 mapper 定义，不替代 `shopflow-db`。

## 2. 复用与依赖边界

- 优先复用入口：`shopflow-core` 中已有的鉴权、存储、通知、支付、任务、公共配置与工具封装。
- 优先复用入口：`shopflow-db` 中已有的 service、mapper、entity、validator 与领域对象。
- 对外依赖边界：允许依赖 `shopflow-core` 与 `shopflow-db`；避免把小程序侧控制器写成底层通用能力承载点。
- 对外暴露边界：对 `shopflow-wx` 小程序端暴露稳定、可兼容的用户侧接口与返回结构。

## 3. 编码约束

- 接口返回结构变更前，必须明确小程序端兼容方案。
- 授权、登录态、购物车、下单、支付、售后、消息相关流程改动时，优先保证用户主路径稳定。
- 输入边界必须校验，不能因为兼容历史前端而静默放开非法参数。
- 用户侧流程中的状态判断应尽量收敛到服务层，避免分散在多个 controller / web 实现中。

## 4. 测试与验证

- 当前目录修改后，优先验证：小程序接口联调、登录授权、下单支付链路、消息与售后流程。
- 高风险点：登录态失效、支付回调、库存与订单状态、优惠券结算、聊天与消息链路。
- 变更后需要同步检查：小程序页面、接口文档、配置项、SQL、支付与通知相关配置。

## 5. OpenSpec 衔接

- 涉及当前目录的功能新增、行为变更、主流程调整时，默认先走 OpenSpec。
- `apply` 完成后自动进入 `review`。
- `review` 完成后停止等待用户确认，再决定是否 `archive`。
- 如果归档结论只影响当前目录，优先沉淀到当前 `AGENTS.md`。
